## What This Is

ConnectivityExplorer is a custom web-based spatial transcriptomics viewer built to phenocopy the 10x Xenium Explorer interface, with first-class support for cell-to-cell connectivity data produced by the lab's R analysis pipelines (initially: NICHESv2). The core motivation is visualizing 488+ ligand-receptor mechanisms (LRMs) as a vector edge layer on top of the Xenium morphology image, with the ability to toggle individual LRMs, filter by strength, and color by expression — none of which Xenium Explorer supports natively.

The tool is deployed via Docker so any lab member can run it locally against their own data, or it can be hosted as a shared lab server.

## Key Technical Notes for Future Sessions

### Coordinate system
- Xenium data (transcripts, cells, boundaries, edges) is stored in **µm**
- Image pixels = µm / pixel_size (typically 0.2125 µm/px)
- The viewer renders everything in image pixel coordinates
- All backend bbox params are in image pixel coords; readers convert internally

### OSD ↔ deck.gl sync (stale closure fix)
- OSD animation handlers capture `syncDeckFromOSD` at init time
- Fix: `syncRef = useRef(null)` updated every render; handlers call `syncRef.current?.(viewer)`

### deck.gl coordinate mapping
- Uses `OrthographicView` with `flipY: true`
- Screen ↔ data: `x = cx + (sx - cW/2) / 2^zoom` where cx/cy is OSD center in image px
- `glOptions={{ preserveDrawingBuffer: true }}` required for PNG screenshot to work

### DZI cache
- Dev mode: cache written to `dataset_path/.dzi_cache/`
- Docker: `CACHE_DIR=/cache` env var redirects to named volume (data mount is `:ro`)
- Pyramid is built on first request, then served from cache indefinitely

### Edge schema tolerance
- `edges.parquet` is optional — if absent, Edges layer is empty but app doesn't crash
- `edgeSchema?.columns` guard prevents crash when schema fetch returns 404

## Step 13 — NICHESv2 Integration

### Context
NICHESv2 is a separate R package in active development that performs cell-to-cell ligand-receptor communication analysis on Xenium spatial data. The goal is for NICHESv2 to output data in the `edges.parquet` format that ConnectivityExplorer already consumes natively — making the two packages work together without any glue code.

### Current state
- ConnectivityExplorer already reads `edges.parquet` with an open schema (any columns beyond x1/y1/x2/y2 are auto-available)
- The `docs/data_format.md` file specifies exactly what NICHESv2 needs to output
- Synthetic test edges are generated by `sample_data/make_edges.py` (k-NN pairing, random LRMs) for UI development

### Tasks for Step 13

**13a — Audit NICHESv2 output schema**
- Determine exactly what columns NICHESv2 produces: column names, dtypes, coordinate space
- Identify any mismatches with the ConnectivityExplorer contract (coordinate system especially)
- Document required vs. produced columns; propose any NICHESv2-side changes

**13b — Write the NICHESv2 → ConnectivityExplorer export function**
- R function (to live in NICHESv2 or a companion package) that takes a NICHESv2 result object and writes `edges.parquet` in the correct format
- Must handle: coordinate conversion (if NICHESv2 uses a different spatial reference), lrm_id assignment (stable integer IDs for the 488 mechanisms), strength normalization

**13c — End-to-end validation**
- Run NICHESv2 on a test Xenium dataset → export → load in ConnectivityExplorer
- Verify edges align correctly with cell centroids and morphology image
- Verify LRM catalogue populates and mechanisms toggle/filter correctly
- Verify strength slider and lrm_expression color mode work with real NICHESv2 strength values

**13d — Consider multiple edge files per dataset**
- The backend already supports an `edge_file` query param (defaults to `edges.parquet`)
- NICHESv2 may produce separate files per analysis run or per condition
- Extend the frontend to allow selecting which edge file to display (dropdown next to the Edges layer toggle)
- Naming convention: `edges_{run_id}.parquet` or `edges_{condition}.parquet`

**13e — Streaming / performance for large NICHESv2 outputs**
- Full NICHESv2 analysis on a large Xenium dataset could produce millions of edges (488 LRMs × many cell pairs)
- Evaluate: DuckDB in-process for server-side spatial queries, or pre-built spatial index (R-tree) on startup
- Target: sub-200ms edge query for 1M+ row files at any zoom level
